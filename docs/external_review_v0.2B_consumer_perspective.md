# Deadman 产品交互完整度评测 v0.2B — C 端用户视角

> 评测日期：2026-06-03  
> 虚拟用户：**重度短剧用户**——番茄/红果/抖音短剧日活，习惯通勤/睡前刷剧，会跳过开头、看到名场面截屏发小红书。不懂技术，不在意 governance，只在意"爽不爽 / 撒不撒糖 / 反转够不够 / 我下次还会不会打开"。  
> 评测目的：v0.2A 是评委视角（5 分钟严苛判断）；本份是 C 端视角，回答"如果上线到番茄 APP，这是个能形成观看习惯的产品吗"。**这份的发现 8 天内基本修不了，主要服务于产品长期路线。**  
> 评测方法：同 v0.2A 的 5 个仪器，但虚拟用户换成短剧重度用户。

---

## 0. TL;DR

**结论：从 C 端短剧用户的视角，这个产品的"互动激发"和"短剧消费心理"之间有结构性错位。Deadman 的产品方向更接近"互动剧场策略游戏"，而不是"短剧增强互动"。**

三个关键不匹配：

| 短剧用户期待 | Deadman 实际提供 | 错位类型 |
|---|---|---|
| 爽点密集（每 30 秒一个） | 1-2 分钟才有 1 个"搭子点" | 节奏 |
| 即时情绪表达（笑哭爽气一键发） | 思考型选择题（"你想怎么分"） | 认知负荷 |
| 看完一部接下一部 | 看一集要分 5 个 episode 切来切去 | 流畅性 |
| 男女主 CP 互动撒糖 | 苦难题材 + 道德抉择 | 题材 |

**这不代表产品方向错——是说"短剧用户"不是它的初始目标人群**。它更可能在 B 站/小宇宙/即刻这类内容深度用户群体里跑得通。

---

## 1. 仪器 1：承诺-兑现地图（C 端视角）

短剧用户看 UI 文案时，期待的是平台习惯（番茄 APP 心智）。Deadman 的文案在评委视角里"有粗粝感是亮点"，在 C 端视角里可能"看不懂、太文艺"。

### 1.1 Catalog 页

| UI 元素 | C 端用户期待 | Deadman 实际 | 错位 |
|---|---|---|---|
| 主标题"要是我来" | 应该像番茄那样直白：剧名 + 集数 + 一个爽点 thumbnail | 抽象命题，没说在干啥 | 入口不直接 |
| Hero 卡 | 应该是男女主 CP 剧照 + 一句撕逼台词 | 灰色背景 + 兔子和锅的简笔画 | 不勾人 |
| "搭子已在场" | 短剧 UI 里没有"搭子"概念 | 内部命名 | 名词陌生 |
| "5 个已发布介入点" | "5 个名场面" / "5 个爽点" / "5 个高能时刻"才是平台话术 | "介入点"是产品经理黑话 | 文案不接地气 |
| "进入" 按钮 | 短剧 APP 一般是直接自动播放第一集 | 要点一下才进 | 多了一步 |

**C 端用户在 catalog 停留判断：5-8 秒**。这 5-8 秒里："这是什么？""为什么我要点？""会不会很烧脑？"——三个问题没一个被回答。

### 1.2 Player 页

| UI 元素 | C 端用户期待 | Deadman 实际 | 错位 |
|---|---|---|---|
| 视频自动播放 | ✅ 自动播 | ✅ | OK |
| 一进来就看到名场面 | 期待第一秒就有冲突/撕逼 | EP12 第一秒是平静对话 | 慢 |
| 弹幕 / 评论入口 | 短剧用户极重度依赖弹幕 | 没有 | 缺核心 |
| "搭子" 角色 | 短剧用户没这个心理预期 | 角落里一个小角色 | 不知是啥 |
| 时间轴 "!" / "?" | 番茄等 APP 没有这种 marker | 抽象符号 | 不解 |
| "火气点 · 你是不是也想说" | 短剧用户会想"你要我说啥" | 半哲学半情绪 | 太重 |
| 选项三选一 | 短剧用户期待的是 1-tap（一个按钮就完事） | 三选一 + 自由输入 | 决策负荷 |
| "我有不同想法" | 短剧用户基本不会用——打字成本太高 | 主推差异化 | 反习惯 |
| 结果"我接住你咯" | 短剧用户期待是"哈哈哈"或者甜蜜瞬间 | 共情型反馈 | 情绪节奏不对 |
| 没有"再来一次/换个选项" | 短剧用户喜欢反复试 | 单线推进 | 缺爽点 |

**关键命中：缺弹幕、缺爽点节奏、缺反复试错入口、缺甜蜜/搞笑情绪反馈**。

### 1.3 自由输入

C 端短剧用户的输入习惯：
- 80% 用户不会主动打字（移动端打字成本高）
- 20% 会打字的用户更喜欢"评论"形态——发出来给别人看，不是"输入然后 AI 回我一句"
- 自由输入 + AI 回应 的双人对话场景在 B 站/即刻可以，但短剧 APP 是 broadcast 媒介

Deadman 的"我有不同想法"在 C 端视角是"用户最不会用的功能位"。

## 2. 仪器 2：Payload 可见度地图（C 端视角）

后端给的 44 字段里，C 端用户**真正想看到**的是另一组（和评委想看的几乎不重叠）：

| 字段 | C 端用户期待 | 当前是否暴露 |
|---|---|---|
| "和别人比"（aggregate_stats） | **高优**——"54% 的人选了 A""我和大家想得一样吗" | ❌ |
| "再来一次"入口 | **高优**——"换一个看看会怎样" | ❌ |
| "分享卡片"（截图友好的结果） | **高优**——发小红书素材 | ⚠️ 按钮存在但无效 |
| "我的爽指数"（可视化分数） | **高优**——单维度爽分，不要多维 | ❌（6 维评分在 payload 但太复杂） |
| "甜蜜/反转/搞笑"标签 | **中优**——剧情情绪分类 | ❌ |
| "下一集预告" | 中优 | ❌ |
| "原剧情对照" | 低优——除非足够爽 | ❌ |
| evidence_refs / applied_constraints | 几乎无价值 | ❌ |
| canon_anchor | 几乎无价值（C 端不在意"我是不是改了 canon"） | ❌ |

**C 端用户想看的 7 项里，Deadman 实装 0 项**。
**评委想看的 8 项里，Deadman 在 payload 里有 6 项（只是不展示）**。

这说明：Deadman 的后端能力是"对评委友好"的，不是"对 C 端友好"的。

## 3. 仪器 3：交互循环 emotional arc（C 端视角）

短剧用户的标准情绪曲线（一集 1-2 分钟）：

```
情绪强度
 ↑
 │       ╱╲          ╱╲        ╱╲
 │      ╱  ╲        ╱  ╲      ╱  ╲
 │     ╱    ╲      ╱    ╲    ╱    ╲
 │    ╱      ╲    ╱      ╲  ╱      ╲
 │   ╱        ╲__╱        ╲╱        ╲
 │  ╱           爽点 2    爽点 3      ╲___ 下集预告
 │ ╱  爽点 1
 ┼─────────────────────────────────────── 时间
```

每 20-30 秒一个爽点（撕逼/打脸/甜蜜/反转），间隔短、密度高。

Deadman 的曲线：

```
情绪强度
 ↑
 │                ╭───╮
 │               ╱     ╲ ← 选完看结果
 │              ╱       ╲
 │             ╱         ╲___ 平
 │  ╱──── 平 ─╲
 │ ╱           ╲ ← 弹出选项面板
 │╱            
 ┼──────────────────────────────────── 时间
 │ 0           60s        90s          1:06
```

**问题**：
1. 单 loop 时长 1-1.5 分钟，C 端用户耐心是 30 秒
2. 唯一情绪峰在"看到结果一行话"——爽度 6/10，远不到短剧爽点的 9/10
3. 没有第二个爽点——选完就完了
4. 跨剧/跨集切换有断点（要回 dropdown）

**C 端用户实际反应**：看 30 秒不知道在干啥 → 划走。**比"看完不爽"更糟糕的是"没看进去"**。

## 4. 仪器 4：盲读截图测试（C 端视角）

把同样 5 张截图给一个 24 岁、每天通勤看短剧的女生看：

### 截图 1：catalog
> "啥？要是我来什么？这是个剧吗？为什么界面这么黑。算了。"
- **流失率**：很高
- **关键问题**：题材"荒年啃树皮"不是 C 端主流偏好（甜宠/总裁 > 苦情）

### 截图 2：player 首屏
> "哦在演古装剧。下面那个橘色小娃娃是啥？她在干嘛。"
- **理解度**：低
- **关键问题**：搭子的存在意义没说，C 端用户不会主动探索

### 截图 3：选项面板
> "等等，这是在让我玩选剧情吗？兔子肉怎么分——我管啊这关我啥事，我就想看剧。"
- **核心错位**：C 端用户的代入感是"看主角爽"，不是"我替主角决定"
- **认知负荷**：高，三选一不是好操作

### 截图 4：结果面板
> "我接住你咯——好可爱。但'这步可以，爽点不是把后面全改了'——这写得太文了吧，谁短剧用户会想看这个。"
- **情绪温度**：太冷静，缺爽点
- **文案**：评委觉得有粗粝感是优势，C 端觉得"绕、不直接"

### 截图 5：episode dropdown
> "为啥要切集？我以为是一部完整剧。每集就 1-2 分钟还要我自己切？算了不看了。"
- **结构错位**：C 端期待的是连续剧 binge，不是 anthology

**盲读结论**：C 端用户大概率在第 2-3 屏流失。不是设计差，是和 C 端短剧消费心理的结构性错位。

## 5. 仪器 5：5 分钟后的 elevator pitch（C 端视角）

让 C 端用户 5 分钟使用后向闺蜜复述：

**乐观情况**：
> "有个看古装剧的 APP，会弹个橘色小娃娃让你选剧情怎么走。挺特别但不太理解。"

**悲观情况**：
> "下载了个看剧 APP 结果不让我安静看剧，老让我做题。卸了。"

**最关键句子**："这能让我下次再打开吗？"
- 答案：**大概率不会**，除非用户本身就喜欢"互动剧"这一垂类

**类比**：Deadman 对短剧用户的关系，类似 Black Mirror Bandersnatch 对普通 Netflix 用户的关系——新鲜感一次性的，不会形成消费习惯。

## 6. C 端视角综合判断

回到用户原始担心：**"产品交互上不足"**——从 C 端视角看，这个不足**不是 UI 问题，是产品定位问题**。

具体表现：
1. 短剧用户的爽点节奏（30 秒/次）和 Deadman 的互动节奏（1-2 分钟/次）差一个量级
2. 短剧用户的情绪表达诉求是 broadcast（弹幕/截屏/转发）而非 dialog（和 AI 对话）
3. 短剧用户的代入感是"看主角"，Deadman 假设代入感是"替主角"——这是 RPG/Visual Novel 心智，不是短剧心智
4. 题材选择（荒年苦难）和短剧 top-loading 题材（甜宠/逆袭/重生）不匹配
5. 自由输入功能在 C 端是低使用率位置，但在 Deadman 是核心差异化——错配

## 7. 8 天内 C 端可见度补救（有限）

**坦诚说：8 天内修不了 C 端定位问题。但有 3 个低成本动作能稍微贴近**：

### 优先级 1：把"分享卡片"做成截图友好的素材（< 4 小时）
- 现在分享按钮无反馈
- 改成生成一张图片卡片（搭子举牌 + 你的选择 + AI 一句回应 + 二维码占位）
- 这是 C 端用户**唯一会反复用的功能**——发小红书需求真实
- 即便答辩用不上，这是产品长期 C 端 vitality 的种子

### 优先级 2：替换 catalog 的题材锚点（如可能，< 8 小时）
- 现在 hero 卡是"荒年啃树皮"这种重题材
- 如果 production pipeline 允许换 episode，挑一个有反转/甜蜜的剪辑作为主推
- huangnian 里有没有相对轻松的桥段？（这取决于 data 可用性）
- 如果不行，至少在 catalog 文案上软化："今天的高光"而不是直接拉冷暗 hero

### 优先级 3：加 1-tap 情绪表达（不替代选项，作为兜底）（< 6 小时）
- 在 marker 命中时，先弹一个轻量的"🔥 来了/💔 心疼/😅 离谱"三选一的小气泡
- 用户点一下完事——和"选剧情"的重决策分层
- 满足"短剧用户即时情绪表达"诉求，靠近宣讲题目最直白的"即时互动激发"

## 8. 长期产品路线建议（不在 8 天范围内）

如果 Deadman 想真正服务 C 端短剧用户，需要的不是 UI 改良而是定位重做。两条路：

### 路线 A：保留"要是我来"内核，换目标人群
- 不打短剧用户，打"互动剧 / Visual Novel / 文字 RPG"用户
- 平台不是番茄红果，是 B 站 / 小宇宙 / 即刻 / 文学站
- 题材换苦情留好（这群用户喜欢深度）
- 这条路 Deadman 现在的产品形态完全成立，只是要换市场叙事

### 路线 B：保留短剧用户人群，换核心机制
- 弹幕化"要是我来"——让"你的选择"变成可看到他人选择的 broadcast
- 加入 1-tap 情绪/反应（爽/哭/笑）作为主线
- "选剧情"作为高级用户的二级功能，不是主推
- 这条路要把 Deadman 当前的 governed runtime 等深度逻辑藏到水下，C 端只看到"快爽轻"

**两条路不可兼得**。当前 demo 是路线 A 的产品形态但用了路线 B 的赛题/平台/题材清单——这是结构性张力。

---

## 9. 给答辩的反向利用

虽然 v0.2B 主要服务长期路线，但有 1 点可以用来强化答辩：

**在答辩 Q&A 时主动声明定位**：
> "我们选择不去做弹幕式即时表达，而是做'你替角色思考一次'的深度互动。这不是为了打短剧大盘用户，是为了证明 AI 能成为剧情参与者，而不是评论生成器。"

这个声明把 v0.2B 发现的"C 端不匹配"翻转成"主动取舍"，反过来抬高 v0.2A 里的差异化分。

## 10. C 端视角分数（参考用，非官方 rubric）

如果有"上线适配度"维度：

| 维度 | 分 |
|---|---|
| 短剧用户首屏吸引力 | 35/100 |
| 短剧用户单次留存（看完 1 个 loop） | 50/100 |
| 短剧用户次日留存（下次还会打开） | 25/100 |
| 短剧用户分享意愿 | 40/100（分享按钮坏了不算） |
| 短剧用户社交传播潜力 | 35/100 |
| **C 端综合（10 分制）** | **3.5/10** |

**这个分不影响答辩——答辩看的是评委视角（v0.2A）**。但产品负责人/路线规划者需要知道这个分，否则上线后会非常痛苦。

---

## 11. 一句话总结

> Deadman 是一个**对评委友好、对 C 端短剧用户错位**的产品形态。8 天内补 v0.2A 的清单能拿好答辩分；C 端化要等下一个版本，且需要在两条不同路线里做选择。
